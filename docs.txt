## 1. Pendahuluan & Paradigma Arsitektur

Project **INFEST USK 2025 Admin** merupakan aplikasi backoffice untuk mengelola pendaftaran peserta, tim, dan administrasi kompetisi Informatika Festival (INFEST) Universitas Syiah Kuala 2025. 

Secara paradigma arsitektural, aplikasi ini menerapkan **Backend-as-a-Service (BaaS)** berbasis **Supabase**. Mayoritas operasi manipulasi database dan manajemen autentikasi diinisiasi langsung dari client-side browser menggunakan SDK Supabase Client (`@supabase/supabase-js`), memanfaatkan Row Level Security (RLS) pada PostgreSQL untuk keamanan.

Namun, untuk proses backend yang membutuhkan privasi credential atau integrasi SMTP mail server, Next.js menyediakan **Route Handler** (server-side endpoint) di bawah `/api/send-notification-email`.

## 5. Daftar Endpoint API & Alur Kerja

Aplikasi Next.js ini memiliki **satu** endpoint custom Route Handler:

### **POST `/api/send-notification-email`**
* **Nama Berkas**: `src/app/api/send-notification-email/route.ts`
* **Keamanan / Auth**: **Public** (Tidak memeriksa Bearer Token, Session Cookie, atau API key).
* **Fungsi**: Mengirimkan email notifikasi status pendaftaran ke email ketua tim peserta.

#### **Alur Kerja Endpoint**:
1. Menerima request POST yang membawa data tim, nama ketua, email, dan status kelulusan (`approved` atau `rejected`).
2. Melakukan validasi dasar (kehadiran field wajib).
3. Melakukan normalisasi URL logo sponsor dan media partner agar kompatibel dengan email client (Google Drive ID parser ke direct download link).
4. Menghasilkan dokumen HTML dinamis dari template email berdasarkan status persetujuan, termasuk menyisipkan tautan WhatsApp group jika status adalah `approved`.
5. Mengirimkan email menggunakan transporter `nodemailer` via Gmail SMTP.
6. Mengembalikan response status sukses atau error dalam bentuk JSON.

## 6. Autentikasi & Otorisasi

Arsitektur autentikasi dalam sistem ini didesain sebagai berikut:

### 1. Proses Autentikasi (Sign-In)
* **Tempat Token Dibuat**: Autentikasi ditangani langsung oleh sistem Supabase Auth di server Supabase saat method `supabase.auth.signInWithPassword` dipanggil di [adminAuthService.ts](file:///home/irfan/Project/infest_admin/informaticsfestival2025_admin-main/src/libs/services/adminAuthService.ts#L20).
* **Penyimpanan Token**: Secara bawaan, SDK `@supabase/supabase-js` menyimpan token session (access token JWT dan refresh token) di `localStorage` pada browser klien.
* **Verifikasi Token**: Setiap query yang dikirimkan ke database Supabase secara otomatis melampirkan JWT di header HTTP `Authorization: Bearer <token>`. Postgres Supabase memverifikasi token ini secara kriptografis untuk mencocokkan identitas pengguna dengan aturan Row Level Security (RLS).

### 2. Otorisasi Peran (Role-Based Access Control)
Terdapat dua level peran admin yang tersimpan di tabel `admin_users`:
1. **`SUPER_ADMIN`**
   * Memiliki akses tak terbatas ke seluruh data registrasi di semua jenis kompetisi.
2. **`ADMIN`**
   * Memiliki akses terbatas yang terikat dengan satu kompetisi tertentu via kolom `admin_competition_id` di tabel `admin_users`.
   * Pengecekan dilakukan secara programatik via helper [isAuthorizedForCompetition](file:///home/irfan/Project/infest_admin/informaticsfestival2025_admin-main/src/libs/helpers/competitionAuthorization.ts#L3) pada halaman kelola kompetisi.

## 7. Middleware & Proteksi Route

Next.js mengandalkan berkas [middleware.ts](file:///home/irfan/Project/infest_admin/informaticsfestival2025_admin-main/middleware.ts) di tingkat root untuk membatasi navigasi halaman pada sisi server:

### 1. Konfigurasi Proteksi
* **Protected Routes (Harus Login)**: `/`, `/users`, `/competitions`, `/teams`, `/settings`.
* **Auth Routes (Hanya untuk tamu)**: `/login` (jika admin sudah login mengakses `/login`, dia akan langsung dilempar ke `/`).

### 2. Mekanisme Pengecekan Token Middleware
Middleware membaca cookie session dengan nama:
`sb-qxesqdjdiuuzhmgadewe-auth-token` (format default token Supabase Auth).
* Jika rute termasuk *Protected Route* dan cookie tersebut **tidak ada**, user dialihkan ke `/login`.
* Jika rute termasuk *Auth Route* (`/login`) dan cookie tersebut **ada**, user dialihkan ke dashboard `/`.

### 3. Masalah Desain Rute API
Konfigurasi matcher mengecualikan rute API:
```typescript
export const config = {
  matcher: ['/((?!api|_next/static|_next/image|favicon.ico|public).*)']
};
```
Karena hal ini, **seluruh endpoint API di `/api/*` dilewati begitu saja oleh middleware**, tidak mendapatkan proteksi autentikasi apapun di tingkat server.

   * `updateCompetitionWhatsappGroup(competitionId, url)` ──> `UPDATE` whatsapp_group ──> `competitions`

5. **Manajemen Tim**
   * `fetchTeams(page, limit, search, status)` ──> `SELECT` view/table ──> View `teams_with_member_count` atau fallback tabel `teams`

## 10. Validasi Request & Response

### Endpoint: `POST /api/send-notification-email`

#### **Request Body (JSON)**
Semua field berikut bertipe `string` dan wajib dikirimkan (kecuali `adminNotes` dan `whatsappGroupLink` yang bersifat opsional):
```json
{
  "teamName": "Tim Antigravity",
  "teamLeaderName": "Irfan",
  "teamLeaderEmail": "irfan@domain.com",
  "competitionName": "UI/UX Design Competition",
  "status": "approved", // "approved" atau "rejected"
  "adminNotes": "Bukti pembayaran valid, selamat bergabung!", // opsional
  "whatsappGroupLink": "https://chat.whatsapp.com/Gxyz..." // opsional, hanya dikirim jika status approved
}
```

#### **Response (Success - HTTP 200)**
```json
{
  "success": true,
  "message": "Email notification sent successfully"
}
```

#### **Response (Validation Error - HTTP 400)**
Terjadi apabila ada field wajib (`teamLeaderEmail`, `teamName`, `competitionName`, atau `status`) yang bernilai kosong atau tidak dikirim.
```json
{
  "error": "Missing required fields"
}
```

#### **Response (Server Error - HTTP 500)**
Terjadi apabila koneksi SMTP gagal atau transporter Nodemailer mengalami kendala crash internal.
```json
{
  "error": "Failed to send email notification"
}
```

## 11. Penanganan Error (Error Handling)

Pola penanganan kesalahan dalam backend project ini mengadopsi prinsip **Graceful Failures**:

### 1. Blok `try-catch` Luas
Setiap fungsi di service layer dibungkus dalam blok `try-catch`. Ketika terjadi kesalahan, sistem tidak membiarkan aplikasi crash (throw error), melainkan menangkap error tersebut, mencatatnya ke konsol log via `console.error`, dan mengembalikan object status gagal untuk ditangani UI:
```typescript
try {
  // Query / Aksi database
} catch (error: any) {
  console.error("Context error:", error);
  return { success: false, error: "Pesan kesalahan yang user-friendly" };
}
```

### 2. Status Code HTTP pada Route Handler
Di route handler server-side (`/api/send-notification-email`), error ditangkap dan dibungkus menggunakan Response JSON Next.js dengan status code yang tepat:
* **`400 Bad Request`**: Kesalahan parameter input / data tidak lengkap.
* **`500 Internal Server Error`**: Kegagalan sistem eksternal (SMTP gagal).

### 3. Toast UI Notification
Hasil error yang dikembalikan oleh service layer ditangkap di halaman antarmuka (Client Components) lalu ditampilkan ke admin dalam bentuk pop-up toast yang interaktif menggunakan library `react-hot-toast` (`toast.error(...)`).

## 12. Layanan Eksternal (External Services)

Backend terintegrasi dengan layanan pihak ketiga berikut:

1. **Supabase (BaaS)**
   * **Database**: PostgreSQL berkapasitas RLS.
   * **Autentikasi**: Supabase Auth OAuth/Password Manager.
   * **Penyimpanan Berkas (Storage)**: Menyimpan berkas bukti pembayaran, proposal, surat orisinalitas, dan bukti twibbon.
2. **Gmail SMTP (Nodemailer Transporter)**
   * Digunakan untuk mengirim email transaksional status registrasi secara asinkron.
3. **Google Drive API (Parser)**
   * Endpoint email secara otomatis mendeteksi URL berkas yang di-upload dari Google Drive (format `/file/d/<id>/`) dan mengonversinya menjadi URL direct download (`/uc?export=view&id=<id>`) agar logo instansi atau file gambar bisa dimuat langsung di aplikasi pembaca email (seperti Gmail, Outlook).

## 13. Flow Diagram Alur Kerja Utama

### 1. Alur Pengiriman Notifikasi Email (`POST /api/send-notification-email`)

```mermaid
sequenceDiagram
    participant Client as Admin Browser (Dashboard UI)
    participant Server as Next.js API Route Handler
    participant Gmail as Gmail SMTP Server

        Server-->>Client: HTTP 400 (JSON Error: Missing required fields)

            Gmail-->>Server: Email terkirim
            Server-->>Client: HTTP 200 (JSON Success: Email sent successfully)
            Gmail->>UserMail: Kirim Email Notifikasi Ke Penerima
        else SMTP Transaksi Gagal (Timeout / Auth Fail)
            Server-->>Client: HTTP 500 (JSON Error: Failed to send email)
        end
    end
```

## 15. Temuan Arsitektur & Keamanan (Findings)

Selama proses scanning dan peninjauan arsitektur kode sumber backend, ditemukan beberapa celah keamanan, kesalahan konfigurasi, dan potensi bug yang perlu segera diperbaiki:

### 1. Paparan Supabase Service Role Key di Sisi Browser (Browser Leak)
* **Tingkat Keparahan**: 🚨 **CRITICAL**
* **Deskripsi**: Variabel `process.env.NEXT_PUBLIC_SUPABASE_ROLE_KEY` memiliki awalan `NEXT_PUBLIC_`. Di Next.js, variabel dengan awalan tersebut secara otomatis diekspos ke browser bundle dan dapat dibaca oleh siapa saja melalui panel devtools browser. Terlebih lagi, pengecekan `typeof window !== 'undefined'` di [supabaseClient.ts](file:///home/irfan/Project/infest_admin/informaticsfestival2025_admin-main/src/libs/services/supabaseClient.ts#L25-L27) sengaja dikomentari, memungkinkan client-side mengeksekusi klien bypass RLS.
* **Dampak**: Penyerang dapat mencuri service role key ini dan menggunakannya untuk membaca, memodifikasi, atau menghapus seluruh tabel database di Supabase tanpa terhalang kebijakan RLS (Row Level Security).
* **Rekomendasi Perbaikan**: 
  1. Hapus awalan `NEXT_PUBLIC_` dari variabel tersebut (ubah menjadi `SUPABASE_SERVICE_ROLE_KEY`).
  2. Aktifkan kembali proteksi error server-only pada fungsi inisialisasi:
     ```typescript
     if (typeof window !== 'undefined') {
       throw new Error('Service role key should not be used in the browser');
     }
     ```

### 2. Endpoint API Kirim Email Tanpa Proteksi Autentikasi (Public SMTP Abuse)
* **Tingkat Keparahan**: 🔥 **HIGH**
* **Deskripsi**: Rute `/api/send-notification-email` adalah endpoint POST publik yang sama sekali tidak memeriksa session token admin maupun API key. Rute ini juga dikecualikan secara eksplisit oleh middleware.
* **Dampak**: Pihak luar (bot/attacker) dapat mengirimkan request spam secara massal ke endpoint ini untuk mengirim email phishing atau spam kepada siapa saja menggunakan SMTP Gmail resmi INFEST. Hal ini dapat menghabiskan kuota harian pengiriman email SMTP dan menandai domain resmi INFEST sebagai penyebar spam (domain blacklisting).
* **Rekomendasi Perbaikan**: 
  Verifikasi sesi pengguna secara server-side di route handler sebelum memproses pengiriman email:
  ```typescript
  import { createServerClient } from "@supabase/ssr";
  // Verifikasi JWT session cookie user di server-side sebelum memproses email
  ```

### 3. Middleware Autentikasi Mengalami Kerusakan Desain (Broken Cookie Sync)
* **Tingkat Keparahan**: 🔥 **HIGH**
* **Deskripsi**: Berkas `middleware.ts` membaca cookie `sb-qxesqdjdiuuzhmgadewe-auth-token` untuk mendeteksi status login. Namun, inisialisasi Supabase client di `adminAuthService.ts` hanya menggunakan SDK standar `@supabase/supabase-js` yang menyimpan token di browser `localStorage`. Tidak ada logika di client-side maupun server-side yang menyalin token ini ke cookie.
* **Dampak**: Admin akan terjebak dalam siklus pengalihan halaman (redirect loop) ke `/login` saat mencoba mengakses halaman dasbor, karena middleware di server-side tidak pernah mendeteksi kehadiran cookie tersebut, meskipun admin sebenarnya sudah sukses login di sisi klien (local storage).
* **Rekomendasi Perbaikan**: 
  Gunakan `@supabase/ssr` untuk menginisialisasi Supabase client di Next.js yang secara otomatis mengelola sinkronisasi token autentikasi ke dalam cookie HTTP-only.

### 4. Skrip Pengaturan SQL Database & CLI Hilang dari Repositori (Missing Scripts)
* **Tingkat Keparahan**: ⚠️ **MEDIUM**
* **Deskripsi**: Script `create-admin` dan `setup-db` di `package.json` merujuk ke berkas di folder `src/libs/scripts/` (seperti `create-admin.ts` dan `admin_users_schema.sql`). Namun, berkas-berkas tersebut tidak ada di repositori.
* **Dampak**: Developer baru tidak dapat melakukan migrasi awal database lokal atau membuat akun administrator pertama via CLI.
* **Rekomendasi Perbaikan**: Kembalikan atau buat ulang file-file skrip tersebut agar developer bisa melakukan setup secara mulus.

### 5. Dependency Terinstal tetapi Tidak Digunakan (Unused Dependencies)
* **Tingkat Keparahan**: ℹ️ **LOW**
* **Deskripsi**: Package `jsonwebtoken` dan `@supabase/ssr` terinstal di dependencies `package.json` tetapi tidak ada berkas kode program yang mengimpornya.
* **Dampak**: Menambah ukuran folder `node_modules` secara tidak perlu.
* **Rekomendasi Perbaikan**: Hapus package tersebut via `npm uninstall jsonwebtoken @supabase/ssr` jika memang sistem autentikasi murni mengandalkan SDK Supabase Client bawaan.
